## Continuous integration * Automatically build and test the software on a regular basis(daily or several times a day). * Notify the developer when the build fails, and include as much detail as needed. * Don't exaggerate with the details + A commit ID(with a direct link in the VCS) should suffice
## Continuous delivery * Next step after CI. * You trust your build process and tests to provide a level of quality that guarantees the software is always in a releaseable state. * More of a philosophy/commitment to ensure that the code is always release-ready. > Customers may not need constant updates.‚Äù
## Continuous deployment * Next step after CI/CD(1). Delivering features/fixes to the customer as soon as they're ready. * By "ready" we understand that all tests passed, build was successful and everything is in shape.

Where to start?

  1. Where are we now?

  2. What do we want?
  3. Why do we want this?
  4. How can we implement this process whilst maintaining some form of control over the deployment process?
  5. What's the next action?

Main C* components

  1. Team culture
  2. Environments
  3. Deployment process
  4. Rollback/recovery process
  5. Development practices
### Team culture * Get everyone involved (Devs, DBAs, Managers) * Outline the investment everyone needs to make and what will they get out of it
### Environments * Get a good environment separation + Dedicated *dev* environments. + Dedicated staging environment(s) identical to production for testing the builds. * Dedicated build environment where all integration takes place.
### Deployment process * Automated build integrity-checks
### Rollback/recovery process * Immediate restore from backup * Pre-tested rollback versions(prev-master) * Fallback environments + Previous version stored until current deploy status turns green
### Development practices * Most difficult to tackle + Personal preferences/practices involved * Modular commits * Automated code validation(linting) * Automated testing * Peer review
## Steps towards CI * Use an issue tracker for everything * Create a separate branch for the work done, tagged with the issue number * Develop on the branch, and continuously test and integrate it * Optional: Push your changes to a rush box(dev box) * When ready, create a pull request for the branch * Merge and release + Deploy to staging + Promote to production
